### What each ministry data file exports

Every file exports a single `MinistryData` object. The `MinistryData` type (from `lib/types.ts`) contains:

```typescript
export interface MinistryData {
  overview: MinistryOverview;     // L0 card + L1 header
  revenue: RevenueData;           // L1 card 3 (null for most ministries)
  fixedObligations: FixedObligationsData;  // L2 bucket 1 + L3 drill-down
  operational: OperationalData;   // L2 bucket 2 + L3 drill-down
  capital: CapitalData;           // L2 bucket 3 + L3 drill-down

}
```

## Extraction playbook: what to extract for each ministry

Every ministry gets the same 9-step treatment. The output for each ministry follows the Finance pattern from `overview.md`.

### Step 0: Historical data extraction (from 2025-26 PDF)

Before touching the 2026-27 document, extract from the **2025-26 Estimates** for this ministry's Heads:

1. **2023-24 Actual (Provisional) expenditure** — at Head level and programme level where available. This is real historical spend.
2. **2024-25 Approved and Revised** — gives budget-vs-revised variance for the year before last.
3. **2025-26 Estimates at programme/sub-programme level** — the current doc only has prior-year totals in summary columns. The detailed 2025-26 pages give you the programme breakdown needed for proper YoY comparison within each bucket.
4. **Capital projects from 2025-26** — project codes, allocations, and any that are continuing into 2026-27.

This data feeds: `priorYearAllocation`, `twoYearsAgoActual`, trend sparklines, and YoY badges.

**Structural mapping:** If the ministry changed structure between years (see the structural differences table above), map old Heads to new Heads. For example, Head 56000 (Science & Technology) in 2025-26 maps to Head 69000 (Energy, Transport & Telecom) in 2026-27.

### Step 1: Ministry overview (from both documents)

Extract from the 2026-27 doc:
- **Total recurrent budget** (from summary TOC, pages 1-6)
- **Total capital budget** (from capital TOC)
- **Programme breakdown** at Head level

Extract from the 2025-26 doc:
- **2025-26 Approved Estimate** (this becomes `priorYearAllocation`)
- **2023-24 Actual** (this becomes `twoYearsAgoActual`)
- **Vision statement** and **Mission statement** (if present — usually on first page of Head)
- **Vision 2030 alignment** (which national goals)

Compute:
- **Year-on-year change**: `(2026_27_estimate - 2025_26_approved) / 2025_26_approved × 100`
- **3-year trend**: `[2023_24_actual, 2025_26_estimate, 2026_27_estimate]`
- **Medium-term projections**: `[2027_28, 2028_29, 2029_30]` from the 2026-27 doc

### Step 2: Budget structure (from recurrent pages)

For the core Head (e.g. 41000 for Education), extract:
- **Programme breakdown:** Each programme code, name, and approved estimate
- **Sub-programme breakdown** within each programme
- **Activity-level detail** for the largest or most significant activities
- **Object classification summary:** Compensation (21), Goods & services (22), Grants (27), Travel (28), Utilities (24), Fixed assets (31/32), etc.
- **Appropriations-in-Aid** if any (self-funded components)

### Step 3: Sub-heads / departments

For each sub-head (e.g. 41010 Jamaica Information Service, 41051 Child Protection):
- **Name and Head number**
- **Total approved estimate** (2026-27 and 2025-26)
- **2023-24 Actual** (from 2025-26 doc)
- **Programme structure** if different from core
- **Key activities and their budgets**

### Step 4: Capital projects

From the capital section (Head XXXXXC), extract for each project:
- **Project code and name**
- **2026-27 allocation**
- **2025-26 allocation** (from prior-year doc for projects that span both years)
- **Total project cost** (if stated)
- **Project description** (usually in narrative)
- **Funding sources** (GOJ, IDB, EU, World Bank, etc.)
- **Timeline** (start/end dates if stated)
- **Forward projections** (2027-28 through 2029-30)

### Step 5: Identify the three buckets

This is the analytical step. Based on the budget structure, classify each programme/activity into:

**Bucket 1: Fixed obligations**
- Look for: statutory payments, pensions, debt service, insurance premiums, membership fees, transfers to public bodies/agencies, grants that are non-discretionary
- Key indicators: the activity description says "contribution to," "transfer to," "subscription/membership," "premium," or the object code is 27 (Grants and Contributions)
- These are pass-through or scheduled payments

**Bucket 2: Operational programmes**
- Look for: divisions/departments with staff doing ongoing work, regulatory functions, service delivery, enforcement
- Key indicators: significant compensation of employees (object 21), use of goods and services (object 22), the entity has a named operational function
- These are the agencies "doing the work"

**Bucket 3: Capital projects**
- Everything in Head XXXXXC
- Discrete projects with project codes, timelines, defined deliverables

**For each bucket, provide:**
- Total allocation (J$) and % of ministry
- Which Head/Programme/Activity codes are included (scope definition)
- Prior year allocation for the same scope
- 2023-24 actual for the same scope (if extractable)

### Step 6: Identify the unique metric

Every ministry has something that makes it distinct. This becomes the third L1 card (where Finance has "Revenue Performance"). The unique metric replaces or supplements the standard three-card layout.

### Step 7: Identify key persons

For each ministry, search the web for (use queries like "[Ministry name] Jamaica minister 2025 2026"):
- **Minister** (political head) — name, title, avatar image
- **State Minister(s)** if any — name, title, avatar image
- **Permanent Secretary** (administrative head) — name, title, avatar image
- **Deputy PS** or equivalent — name, title
- **Heads of sub-head entities** (Commissioner, Director General, Chief Medical Officer, etc.) — name, title

**Output format for each person:**
```typescript
{
  name: 'Hon. Full Name, MP',
  title: 'Minister of X',
  role: 'minister' | 'state_minister' | 'ps' | 'deputy_ps' | 'head_officer' | 'director',
  headCode: '20000',  // for entity heads
  avatarUrl: '/avatars/slug.jpg',  // placeholder initially
}
```

**Avatar handling:** Initially use `/avatars/placeholder.svg`. When real images are sourced, save to `/public/avatars/{first-last}.jpg`. The people data file should be immediately usable with placeholders.

### Step 8: Define the dashboard structure (levels 0-3)

This is the key analytical output. For each ministry, define what appears at each level — exactly the same structure as the Finance spec in `overview.md`.

#### Level 0 — Cabinet card

Every ministry card on the cabinet grid shows:
- Ministry name
- Minister avatar + name
- Total allocation (single number)
- Budget utilization % with progress bar (mock data only)
- Status badge (on-track / at-risk / off-track)
- YoY badge vs prior year

#### Level 1 — Ministry summary (3 metric cards)

| Card | Content | Notes |
|------|---------|-------|
| Card 1: Total Allocation | Total approved (recurrent + capital), YoY change, recurrent/capital split | Universal for all ministries |
| Card 2: Budget Utilization | Utilization %, expected %, utilization gauge | Universal for all ministries |
| Card 3: Unique Metric | Varies by ministry (see Step 6 table) | Finance = Revenue, Education = Enrollment, etc. |

**Most ministries will NOT have a revenue card.** For them, Card 3 becomes the unique metric OR a staffing summary.

#### Level 2 — Three bucket summary cards

| Card | Content | Universal? |
|------|---------|------------|
| Fixed Obligations | Total allocation, total paid, % of ministry, obligations count, all-current badge | Yes — every ministry has some fixed obligations (grants, transfers, insurance at minimum) |
| Operational Programmes | Total allocation, spent, utilization %, entity count, vacancy rate | Yes — every ministry has operational divisions |
| Capital Projects | Total allocation, spent, project count, on-track count, avg physical progress | Only if ministry has capital budget (most do) |

#### Level 3 — Drill-down cards

**Fixed obligations cards** — one per obligation type. Common types across ministries:
- Grants & contributions to public bodies
- Insurance premiums (health insurance, catastrophe)
- Transfers to agencies
- Statutory payments
- (Finance only: debt amortisation, debt interest, pensions)

**Operational programme cards** — one per entity/department. Each card has:
- Allocation, spent, utilization %
- Staffing: approved posts, filled, vacancy rate
- KPIs: 2-4 entity-specific KPIs with target vs actual
- Spend trajectory chart (mock actuals)
- Entity head (key person)

**Capital project cards** — one per project. Each card has:
- Current year budget and spent
- Total project cost and cumulative spend
- Financial and physical progress %
- Start/end dates, revised dates
- Status + risk level
- Narrative
- Milestones timeline
- Funding sources with disbursement status

### Step 9: Generate the data file

Transform all extracted data into a TypeScript file that conforms to `MinistryData`. This is the deliverable.

**The data file must include:**

1. **All allocation/budget numbers** — from the Estimates documents (both years)
2. **Prior year allocations** — for every entity, obligation, and project
3. **2023-24 actuals** — where available from the 2025-26 doc
4. **Medium-term projections** — [2027-28, 2028-29, 2029-30]
5. **Mock actuals** — simulated 6-month time series (April–September 2026) at ~48-52% utilization
6. **Prior year mock actuals** — simulated 12-month time series for 2025-26
7. **Key persons** — minister, PS, entity heads with placeholder avatars
8. **KPI targets** — derived from Estimates narrative or reasonable defaults
9. **Staffing data** — from Estimates staffing tables
10. **Bucket classification** — which Heads/programmes/activities go in which bucket

**Mock data generation rules:**
- Overall utilization targets: 48-52% at 6-month mark
- Use weight distributions for realistic spending patterns (W_LIN for even, W_FRONT for early-heavy, W_BACK for back-loaded)
- Insurance premiums should be 100% paid (annual upfront payment)
- Capital projects vary: some on track, one delayed, contingency untouched
- At least one KPI should be "at risk" per operational entity for realism

## National / Government-Wide Data Layer

### Current state (hardcoded, `lib/data/national.ts`)
Macro-economic and social indicators sourced from PIOJ quarterly briefings, STATIN releases, and official GOJ reports. Updated manually when new data is published.

### DB migration schema (for Phase 2 FMIS/DB migration)

```sql
CREATE TABLE national_metrics (
  id TEXT PRIMARY KEY,           -- e.g. 'gdp_growth', 'unemployment_rate'
  label TEXT NOT NULL,
  category TEXT NOT NULL,         -- 'macro_economic' | 'social' | 'disaster'
  unit TEXT NOT NULL,             -- 'percent' | 'currency_jmd' | 'currency_usd' | 'ratio' | 'index' | 'rate' | 'count'
  format TEXT,                    -- optional display hint e.g. '% of GDP'
  source TEXT NOT NULL,
  source_url TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE national_metric_values (
  id SERIAL PRIMARY KEY,
  metric_id TEXT REFERENCES national_metrics(id),
  period TEXT NOT NULL,           -- e.g. 'Oct–Dec 2025', 'January 2026', '2024', 'FY 2025/26'
  value NUMERIC NOT NULL,
  provisional BOOLEAN DEFAULT FALSE,
  as_of DATE NOT NULL,            -- when this value was published
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(metric_id, period)
);

CREATE TABLE disaster_events (
  id TEXT PRIMARY KEY,             -- e.g. 'hurricane_melissa'
  name TEXT NOT NULL,
  event_date DATE NOT NULL,
  category TEXT NOT NULL,          -- e.g. 'Category 5 Hurricane'
  total_damage BIGINT NOT NULL,    -- in JMD
  damage_unit TEXT DEFAULT 'currency_jmd',
  gdp_pct_impact NUMERIC,
  description TEXT,
  source TEXT NOT NULL,
  source_url TEXT,
  as_of DATE NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE disaster_sector_impacts (
  id SERIAL PRIMARY KEY,
  disaster_id TEXT REFERENCES disaster_events(id),
  sector TEXT NOT NULL,
  damage BIGINT NOT NULL,
  losses BIGINT NOT NULL,
  UNIQUE(disaster_id, sector)
);
```

### Proposed DB schema for migration

```sql
CREATE TABLE vision2030_goals (
  id INTEGER PRIMARY KEY,          -- 1-4
  name TEXT NOT NULL
);

CREATE TABLE vision2030_outcomes (
  id INTEGER PRIMARY KEY,          -- 1-15
  goal_id INTEGER NOT NULL REFERENCES vision2030_goals(id),
  name TEXT NOT NULL,
  sdgs INTEGER[] NOT NULL DEFAULT '{}'
);

CREATE TABLE vision2030_indicators (
  id TEXT PRIMARY KEY,              -- e.g. '1-1', '7-5'
  outcome_id INTEGER NOT NULL REFERENCES vision2030_outcomes(id),
  name TEXT NOT NULL,
  unit TEXT NOT NULL,
  direction TEXT NOT NULL CHECK (direction IN ('higher_is_better', 'lower_is_better')),
  baseline_2007 NUMERIC,
  target_2027 TEXT,                 -- TEXT to handle range targets like '4.0-6.0'
  target_2030 TEXT,
  source TEXT,
  responsible_ministries TEXT[] NOT NULL DEFAULT '{}',
  discontinued BOOLEAN DEFAULT FALSE,
  note TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE vision2030_actuals (
  id SERIAL PRIMARY KEY,
  indicator_id TEXT NOT NULL REFERENCES vision2030_indicators(id),
  period TEXT NOT NULL,             -- e.g. '2024', 'January 2026', 'FY 2025/26'
  value NUMERIC NOT NULL,
  provisional BOOLEAN DEFAULT FALSE,
  source TEXT,
  recorded_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(indicator_id, period)
);
```

### Backfill plan
1. **Phase 1 (done)**: Hardcoded data from MTF 2024-2027 document. Targets and baselines complete.
2. **Phase 2**: Populate `latestActual` for all indicators with publicly available data from PIOJ, STATIN, World Bank WGI, UNDP HDI, WEF indices. Priority: Goal 3 (economy) and Goal 2 (security/governance).
3. **Phase 3**: Migrate to DB. Seed from hardcoded data. Add admin UI for PIOJ staff to update actuals quarterly.
4. **Phase 4**: Historical time-series — backfill actuals from 2009-2025 using Vision 2030 Jamaica Monitoring Dashboard (data4development.gov.jm) and PIOJ quarterly/annual reports.
